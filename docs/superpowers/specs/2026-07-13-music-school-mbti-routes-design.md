# Music, School 카테고리 및 MBTI 라우트 설계

## 목표

기존 `library`, `nakdong` 이미지 관리 채널에 `music`, `school`을 추가한다. 두 채널은 기존의 공통 이미지 관리 기능을 그대로 사용하며 하나의 대구 관리자 계정을 공유한다. 별도로 `GET /mbti/[type]` 라우트를 추가해 경로 파라미터를 그대로 돌려준다.

## 카테고리 구조

- 허용 카테고리는 `library`, `nakdong`, `music`, `school` 네 가지다.
- 기존 동적 페이지와 API인 `/[category]/admin`, `/[category]/admin/usage`, `/[category]/[uid]`, `/api/[category]/*`를 네 카테고리가 공유한다.
- `music`, `school`의 관리자 화면 제목은 각각 `음악`, `학교`로 표시한다.
- `music`, `school`의 공개 이미지 화면은 기존 `library`의 기본 뷰어를 사용한다.
- `nakdong`의 검은 배경 전체 너비 뷰어는 변경하지 않는다.
- 허용 목록에 없는 카테고리는 기존과 같이 거부한다.

## 인증과 환경변수

- `library`는 기존 `LIBRARY_ADMIN_ID`, `LIBRARY_ADMIN_PASSWORD`를 사용한다.
- `nakdong`은 기존 `NAKDONG_ADMIN_ID`, `NAKDONG_ADMIN_PASSWORD`를 사용한다.
- `music`과 `school`은 공통 `DAEGU_ADMIN_ID`, `DAEGU_ADMIN_PASSWORD`를 사용한다.
- 세션은 기존처럼 카테고리 값을 포함한다. 따라서 같은 관리자 자격증명을 쓰더라도 `music` 세션이 `school` 관리자 화면을 열어 주지는 않는다.
- 로컬 개발 기본값과 README 환경변수 예시에도 대구 관리자 계정을 추가한다.

## 데이터베이스

신규 설치용 테이블 정의의 category CHECK 제약을 네 카테고리로 확장한다. 이미 운영 중인 D1 데이터베이스에는 새 마이그레이션을 적용한다.

SQLite는 기존 CHECK 제약을 직접 수정할 수 없으므로 마이그레이션은 `images`와 `usage_records` 테이블을 새 제약으로 재생성하고 기존 데이터를 복사한다. 기존 열, 데이터, 고유 인덱스와 조회 인덱스를 모두 보존한다.

## MBTI GET 라우트

- 경로: `GET /mbti/[type]`
- 인증: 없음
- 성공 응답: HTTP 200 JSON `{ "type": "<입력된 type>" }`
- Next.js가 URL 경로를 해석해 제공한 `type` 문자열을 대소문자 변환, MBTI 값 검증 또는 별도 정규화 없이 반환한다.
- 이번 범위에는 POST, 저장소, 비즈니스 로직 또는 MBTI 유형 검증을 포함하지 않는다.

## 오류 처리

- `music`, `school`은 정식 카테고리이므로 기존 카테고리 API의 오류 규칙을 그대로 따른다.
- 잘못된 카테고리는 기존 404 또는 API의 잘못된 카테고리 응답을 유지한다.
- `/mbti`처럼 `type` 세그먼트가 없는 요청은 동적 라우트와 일치하지 않으므로 Next.js 404를 사용한다.

## 테스트와 완료 기준

구현은 테스트 우선으로 진행한다.

1. 카테고리 파서가 `music`, `school`을 허용하고 UI 라벨을 노출하는 테스트를 먼저 추가한다.
2. 두 카테고리가 동일한 대구 관리자 자격증명을 선택하되 세션 범위는 분리되는 테스트를 추가한다.
3. 이미지 관리 API가 새 카테고리를 허용하는 대표 테스트를 추가한다.
4. `GET /mbti/[type]`이 입력 문자열을 JSON으로 반환하는 라우트 테스트를 추가한다.
5. 신규 DB 정의와 마이그레이션이 네 카테고리를 허용하고 기존 데이터를 보존하도록 검증한다.
6. 전체 단위 테스트, 타입 검사, 린트와 빌드를 통과시킨다.

완료 후 `music`과 `school`은 기존 채널과 같은 이미지 관리 기능을 제공하고, 두 채널의 관리자 로그인은 대구 공용 계정을 사용한다. MBTI 라우트는 입력된 경로 문자열만 반환한다.
