# Music 오디오 업로드 및 재생 설계

## 목표

`music` 카테고리를 이미지 대신 MP3와 WAV 음악 파일을 관리하는 채널로 변경한다. 업로드 과정에서 이미지 썸네일을 만들지 않고, 관리자 목록과 공개 보기 페이지에서 원본 오디오를 재생하고 다운로드할 수 있게 한다.

`library`, `nakdong`, `school`은 현재 이미지 업로드, 썸네일, 관리자 목록과 공개 이미지 뷰어 동작을 그대로 유지한다.

## 업로드 형식과 검증

- `music`은 MP3와 WAV 파일만 허용한다.
- MP3는 `.mp3` 확장자와 `audio/mpeg` MIME을 사용한다.
- WAV는 `.wav` 확장자와 `audio/wav` 또는 `audio/x-wav` MIME을 사용한다.
- 업로드 MIME이 비어 있거나 `application/octet-stream`이면 파일 확장자로 `audio/mpeg` 또는 `audio/wav`를 보정한다.
- 확장자를 기준으로 허용 형식을 결정하며, 명시된 MIME이 확장자와 맞지 않으면 HTTP 415로 거부한다.
- `music`에 이미지나 다른 오디오 형식이 들어오면 HTTP 415를 반환한다.
- 나머지 카테고리는 기존처럼 `image/*`만 허용한다.
- 파일 누락, 인증 실패와 UID 충돌 재시도는 기존 오류 규칙을 유지한다.

기존 multipart 파서는 이미지 전용 MIME 추론 함수를 미디어 파일 추론 함수로 확장한다. MP3와 WAV 보정을 추가하되 기존 JPG, PNG, WebP, GIF 보정은 유지한다.

## 저장과 썸네일

`music` 업로드는 기존 D1 이미지 메타데이터 테이블과 R2 키 규칙을 그대로 재사용한다. DB 스키마나 마이그레이션은 추가하지 않는다.

- 원본 오디오 Blob은 기존 `images/music/{uid}/{filename}` 위치에 저장한다.
- `music`일 때는 Cloudflare Images 썸네일 생성기를 `createImage`에 전달하지 않는다.
- 따라서 썸네일 생성 API를 호출하지 않고 `thumbnailKey`는 `null`로 저장한다.
- 사용 기록, 만료 시간, 삭제와 정리 작업은 기존 공통 로직을 사용한다.
- 원본 파일 API는 R2 Blob의 `audio/mpeg` 또는 `audio/wav` Content-Type을 그대로 반환한다.

## 관리자 목록

`music` 관리자 목록은 썸네일 대신 재생 UI를 표시한다.

- 첫 번째 열 제목은 `듣기`다.
- 각 행에 `듣기` 버튼을 표시한다.
- 버튼을 누르면 해당 행 안에 브라우저 기본 `<audio controls>` 플레이어가 펼쳐진다.
- 플레이어 소스는 `/api/music/images/{uid}/file`이다.
- 한 번에 한 행만 펼치며, 열린 행의 버튼을 다시 누르면 플레이어를 닫는다.
- 자동재생은 사용하지 않는다.
- UID, 생성시간, 만료시간, 삭제와 페이지 이동 기능은 그대로 유지한다.

`library`, `nakdong`, `school` 목록은 기존 썸네일 열과 이미지 보기 링크를 유지한다.

## 공개 보기 페이지

`/music/{uid}`는 이미지 대신 오디오 전용 화면을 렌더링한다.

- 브라우저 기본 `<audio controls>` 플레이어를 표시한다.
- 소스는 `/api/music/images/{uid}/file`이며 자동재생하지 않는다.
- 파일 UID와 생성시간을 표시한다.
- 기존 원본 다운로드 API를 연결한 `원본 다운로드` 버튼을 표시한다.
- 존재하지 않거나 다른 카테고리의 UID는 기존처럼 404다.

`nakdong`의 검은 배경 전체 너비 이미지 화면과 `library`, `school`의 기본 이미지 화면은 변경하지 않는다.

## 오류와 호환성

- 브라우저가 재생할 수 없는 손상된 MP3/WAV는 업로드 MIME과 확장자 검증을 통과할 수 있으나 플레이어에서 재생 오류로 표시된다. 이번 범위에는 오디오 내용 디코딩 검증을 포함하지 않는다.
- 기존에 `music`으로 저장된 이미지 행이 있다면 새 목록과 공개 페이지에서는 오디오로 취급된다. 데이터 자동 변환이나 삭제는 수행하지 않는다.
- 외부 업로드 API 경로와 성공 응답 형식은 변경하지 않는다.

## 테스트와 완료 기준

구현은 테스트 우선으로 진행한다.

1. `music` MP3/WAV 업로드가 성공하고 MIME 보정이 적용되는 테스트를 추가한다.
2. `music` 이미지와 지원하지 않는 오디오 업로드가 415인 테스트를 추가한다.
3. `music` 업로드가 썸네일 생성기를 호출하지 않고 `thumbnailKey: null`과 원본만 저장하는지 검증한다.
4. 기존 이미지 카테고리의 업로드 및 썸네일 테스트가 그대로 통과하는지 확인한다.
5. 관리자 목록에서 music에만 듣기 버튼과 단일 인라인 오디오 플레이어가 표시되는지 검증한다.
6. 공개 music 페이지가 오디오 플레이어와 다운로드 링크를 렌더링하는지 검증한다.
7. 전체 단위 테스트, 타입 검사, 린트, 프로덕션 빌드와 E2E를 통과시킨다.
